# 00 - Documents Pipeline

Этот файл задаёт порядок чтения и работы с `.md` документами в `leadlag-lab`.

Служебные имена `README.md`, `AGENTS.md` и `plan.md` оставлены без переименования намеренно:

- `README.md` — стандартная точка входа для репозитория.
- `AGENTS.md` — стандартная инструкция для следующих агентов.
- `plan.md` — пользователь сам называет его "файл план"; это главный источник истины.

Остальные рабочие документы пронумерованы. Номера показывают маршрут чтения, но не отменяют приоритет источников истины.

Если документы конфликтуют, приоритет такой:

1. `plan.md` — исходная задача пользователя.
2. `05_PRE_FLUTTER_STABILIZATION_PLAN.md` — что реально делаем до Flutter.
3. `06_FLUTTER_UI_REWORK_PLAN.md` — что переносим в Flutter.
4. `03_NEXT_ITERATION_REVIEW_TO_10.md` — основной аудит для триажа.
5. `04_DEEP_AUDIT_REPORT.md` — дополнительный внешний аудит.
6. `02_FULL_REWORK_PLAN_TO_10.md`, `90`, `91`, `92` — reference/archive.

## Как Работать Дальше

Текущий этап: **05 — pre-Flutter stabilization**.

Не надо одновременно реализовывать все пункты из `02`, `03` и `04`. Это создаст кашу и цикл бесконечного ремонта старого HTML UI.

Правило:

- Если пункт есть в `05_PRE_FLUTTER_STABILIZATION_PLAN.md`, он делается перед Flutter.
- Если пункт есть только в `03` или `04`, он сначала проходит триаж и либо попадает в `05`, либо переносится в `06`.
- Если пункт относится к красоте/полировке старого HTML UI, он почти всегда переносится в `06`.
- Старый HTML UI до Flutter должен быть честным и диагностическим, но не финальным.

## Главная Логика

1. Не строить Flutter поверх сломанного backend/API.
2. Не доводить старый HTML UI до идеала перед Flutter.
3. До Flutter чинить только:
   - correctness;
   - API/data contracts;
   - ложные UI-состояния;
   - broken core workflow;
   - минимальный demo-ready путь.
4. Всё, что связано с новой визуальной иерархией, красивыми таблицами, финальной навигацией и глубоким UX, проектируется в `06_FLUTTER_UI_REWORK_PLAN.md`.

## Acceptance Gate Между 05 И 06

Flutter можно начинать, когда:

- raw parquet можно превратить в analyzed session через UI/API;
- Explorer открывает события;
- baseline strategy существует;
- baseline backtest запускается из UI;
- Trade Inspector открывает сделку;
- Monte Carlo не вводит в заблуждение дефолтным методом;
- collector status не врёт при stale state;
- network chart не сломан;
- Explorer по умолчанию не скрывает события follower-фильтром;
- есть тесты на ключевые P0.

Не надо ждать:

- идеального legacy UI;
- полного Strategy editor в HTML;
- идеального Backtest comparison в HTML;
- полного mobile polish в HTML;
- финальной Flutter-навигации.
