# Обсуждение архитектуры сигналов (2026-04-19)

## Контекст

Leadlag-lab обнаруживает lead-lag арбитражные возможности между криптобиржами.
Текущая система: коллектор → сырые тики → бины 50мс → VWAP → EMA → порог 2.0σ → события → классификация A/B/C.
Проблема: архитектура перевёрнута — инфраструктура решает логику сигналов, стратегия только фильтрует.

---

## Согласованные решения

### 1. Стратегия = мозг, инфраструктура = доставка + исполнение
- Стратегия решает: метод биннинга, сглаживание (EMA/EEMA/иное), порог, логику A/B/C, управление позициями
- Инфраструктура обеспечивает: поток тиков, BBO, исполнение ордеров, учёт позиций, библиотеку утилит

### 2. Бины — это нормально и полезно
- Ускорение бэктеста
- Искусственная задержка в бэктесте симулирует реальную латентность (это ХОРОШО)
- Анализ по EMA даёт предварительную оценку, полезно при добавлении новых бирж
- Трюк «последний тик = закрытие бина» решает задержку в реалтайме — мгновенная реакция на резкий рывок

### 3. Текущий baseline-анализ остаётся
- Бины + EMA + порог + A/B/C — инструмент разведки/исследования
- Не заменяется — полезен для первичного анализа данных
- Эксплорер показывает A/B/C отдельно (нравится пользователю)

### 4. Позиции обязательны
- Стратегия обязана давать сигнал на ЗАКРЫТИЕ, не только на открытие
- Полный жизненный цикл позиции

### 5. Два сосуществующих режима (не замена одного другим)
- Режим 1: Разведка/baseline-анализ (текущий подход, бины+EMA, A/B/C)
- Режим 2: Стратегия (полный контроль, своя логика детекции, свои параметры)

## Нерешённые вопросы

### В1: Интерфейс стратегии — on_tick или on_bar?
- Сначала сошлись что бины работают везде (tick-режим не нужен)
- Но потом вопрос про бины по объёму, адаптивные бины → метод биннинга выбирает стратегия
- Если стратегия выбирает метод биннинга, она всегда получает on_bar()? Или сырой on_tick()?
- **Не решено**: точная сигнатура интерфейса

### В2: Как совместить скорость бэктеста со свободой стратегии?
- Tick-by-tick бэктест ~100x медленнее чем по бинам
- Стратегия может захотеть нестандартный биннинг (по объёму, адаптивный) — нельзя предрасчитать
- **Не решено**: как держать бэктест быстрым при полном контроле стратегии

### В3: Разделение в UI — стратегия vs разведка
- Пользователь явно сказал: текущий интерфейс будет вводить в заблуждение — «где стратегия, где набросок?»
- Должно быть чётко разделено — не просто метки, а отдельные экраны/секции
- **Не решено**: точный подход к UI (вкладки? отдельные страницы? другое визуальное решение?)

### В4: Выбор метода биннинга
- По времени (50мс) — текущий
- По объёму — закрытие при накоплении X BTC
- По количеству тиков — закрытие каждые N тиков
- Адаптивный — в зависимости от волатильности
- Стратегия выбирает, но инфраструктура должна поддерживать несколько методов
- **Не решено**: сколько методов поддерживать, как стратегия объявляет свой выбор

### В5: Классификация A/B/C в режиме стратегии
- В режиме разведки: инфраструктура классифицирует (OKX соло=A, Bybit соло=B, оба=C)
- В режиме стратегии: стратегия решает что считать подтверждением
- Но пользователю нравится видеть A/B/C отдельно в эксплорере
- **Не решено**: стратегия выдаёт теги A/B/C или сырые сигналы? Как эксплорер их показывает?

### В6: Бэктест стратегии
- Запуск strategy.on_bar() по историческим тиковым данным
- Стратегия может использовать другой биннинг → пересчитывать на лету?
- Grid search со стратегией — у стратегии свои параметры, как делать перебор?
- **Не решено**: API бэктеста для стратегий

### В7: Разогрев / холодный старт
- EMA нужно ~200 бинов для разогрева
- Стратегия начинает paper trading → нет сигналов первые ~10 секунд?
- Предзагрузка последних тиков для разогрева?
- **Не решено**: стратегия разогрева

## Аргументы Claude

- Tick-by-tick на Python ~100x медленнее бинов для бэктеста (проблема производительности)
- Разрыв notebook → paper опасен (разные пути кода)
- Если инфраструктура делает детекцию — она ограничивает стратегию, каждый захардкоженный параметр обрезает руки
- Бины с «последний тик = закрытие» дают всё что нужно — не нужен raw tick интерфейс
- Подтверждение C-сигнала: A+B в окне = C, просто, без асинхронной сложности
- Rolling VWAP на каждом тике = нулевая искусственная задержка в реалтайме
- Стратегия должна использовать один и тот же класс в ноутбуке, бэктесте, paper, live — один путь кода
- Режим исследования и режим стратегии должны быть двумя разными вещами в UI
